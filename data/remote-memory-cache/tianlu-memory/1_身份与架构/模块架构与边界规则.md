# 模块架构与边界规则

**创建时间**: 2026-04-22

---

## 模块职责边界

### 控制层（只发指令，不改机器人）

| 模块 | 职责 | 生效方式 |
|------|------|---------|
| **天眼** | 持仓卫士：监控所有持仓，给纠错/止盈/止损指令 | 重启 console_server 生效 |
| **出山** | 出场AI：管理机器人持仓的出场决策 | 重启 console_server 生效 |

### 数据层（采集、对接数据源）

| 模块 | 职责 | 生效方式 |
|------|------|---------|
| **回测** | 机器人的历史数据源 + 策略回测 | 全部配置完毕后再对接机器人 |

---

## 铁律：不要随便重启和修改机器人参数

**绝对禁止：**
- 随意重启正在运行的机器人进程
- 随意修改机器人的交易参数（杠杆/止损/止盈等）
- 通过天眼/出山以外的途径干预机器人

**正确方式：**
- 天眼/出山通过 API 发指令，机器人被动接受
- 机器人本来在跑就不要动
- 改 console_server.py 后只重启 console_server，不动机器人

---

## 模块对接顺序

```
Phase 1: 控制层对接
    天眼 → 机器人（只发指令）
    出山 → 机器人（只发指令）
    → 验证有效后进入 Phase 2

Phase 2: 数据层对接
    回测 + 其他数据采集模块全部配置完毕
    → 再统一对接机器人

Phase 3: 完整联调
    所有模块就位，机器人正常交易
```

---

## 为什么这样设计

- **机器人稳定性**：机器人进程一旦启动，应保持长期运行不受干扰
- **控制层解耦**：天眼/出山是"大脑"，机器人是"手脚"，大脑通过 API 指令手脚，不直接修改手脚的代码
- **数据层可控**：数据采集模块配置完整后一次性对接，避免边改边接导致参数漂移
