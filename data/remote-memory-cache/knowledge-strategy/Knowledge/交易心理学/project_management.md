### 学习任务1：任务优先级管理

#### 四象限法则（紧急×重要矩阵）

| | 紧急 | 不紧急 |
|---|---|---|
| **重要** | 第一象限：危机、紧急任务 | 第二象限：规划、预防、关系建设 |
| **不重要** | 第三象限：干扰、某些会议 | 第四象限：琐事、浪费时间 |

**核心原则：把主要精力放在第二象限（重要不紧急），避免陷入第一象限的救火模式。**

#### 优先级分类标准

| 优先级 | 定义 | 响应时间 | 示例 |
|---|---|---|---|
| **P0** | 立即处理，关乎生死 | 立即，5分钟内 | 系统宕机、重大数据泄露、黑天鹅事件 |
| **P1** | 今日必须完成 | 当日内 | 关键里程碑延误、核心功能bug |
| **P2** | 本周内完成 | 3-5天内 | 重要功能迭代、常规优化 |
| **P3** | 待定，可推迟 | 灵活 | 体验优化、探索性需求 |

#### 任务依赖关系管理

- **阻塞关系**：A任务未完成，B任务无法开始
- **并行关系**：A和B可同时进行
- **依赖关系**：A完成后B开始
- **工具推荐**：使用甘特图或看板管理依赖链

### 学习任务2：跨部门协调

#### 各部门核心诉求分析

| 部门 | 核心诉求 | 关注点 |
|---|---|---|
| **户部（财务/商务）** | 盈利、ROI、成本控制 | 收益、成本、现金流 |
| **兵部（风控/法务）** | 合规、风险最小化 | 风险敞口、违规成本 |
| **工部（技术/运维）** | 系统稳定、技术可行 | 稳定性、扩展性、技术债务 |

#### 跨部门协调流程

```
1. 发起：明确需求和预期结果
2. 沟通：了解对方部门的利益点和约束
3. 协商：找到双赢方案或优先级排序
4. 确认：书面记录agreed的deadline和交付物
5. 跟踪：定期check-in，避免偏离
6. 验收：按时验收，反馈闭环
```

#### Deadline设置原则

1. **明确可衡量**：不是"尽快"而是"3月31日18:00前"
2. **留有Buffer**：关键任务预留20%时间buffer
3. **考虑依赖**：确认上下游可交付时间
4. **早承诺少交付**：承诺保守，交付超预期

### 学习任务3：汇报机制

#### 标准日报格式

```
## 每日简报 | [日期]

**项目：[项目名称]**

| 任务 | 状态 | 完成度 | 风险/阻塞 |
|---|---|---|---|
| 任务A | ✅ 完成 | 100% | 无 |
| 任务B | 🔄 进行中 | 60% | 等待设计稿 |
| 任务C | ⏳ 待开始 | 0% | 无 |

**明日计划：**
- 任务D
- 任务E

**风险提示：**
- [如无则填"无"]
```

#### 立即上报情况（ escalation criteria）

当发生以下情况时，需立即（15分钟内）上报：

| 触发条件 | 上报对象 | 说明 |
|---|---|---|
| 🔴 黑天鹅事件 | 尚书令/相关大臣 | 无法预见的重大事件 |
| 🔴 系统故障/宕机 | 技术负责人+尚书令 | 业务中断 |
| 🔴 重大亏损/超预算 | 户部+尚书令 | 财务影响超阈值 |
| 🟡 关键里程碑延误>2天 | 项目VP | 影响整体交付 |
| 🟡 核心人员变动 | HR+项目VP | 资源风险 |

## 方法总结：任务管理三板斧

### 1. 优先级决策树
```
问：这个任务失败会影响什么？
├── 不影响业务 → P3
├── 影响单日目标 → P2
├── 影响本周目标 → P1
└── 影响核心KPI/生死 → P0
```

### 2. 协调沟通模板
```
[背景]：由于[原因]，需要[你的需求]
[对方利益]：这对[对方部门]的好处是[xxx]
[我的承诺]：我们会确保[xxx]
[请求确认]：请问[时间]前是否可以完成？
```

### 3. 汇报黄金法则
- **及时**：不要等到最后一刻才汇报坏消息
- **简洁**：30秒说清楚问题和建议
- **诚实**：如实反映风险，不报喜不报忧
- **方案**：带着问题+建议方案上报
