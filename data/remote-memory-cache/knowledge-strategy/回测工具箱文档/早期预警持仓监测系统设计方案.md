## 二、现有预警系统分析

### 2.1 检测逻辑（已实装）

位于 `console_server.py` 的 `_detect_wrong_direction` 和 `_detect_premature_entry`：

**方向错误检测（5信号，满足≥2条触发）：**
1. RSI 进入极端区 (>70 做多 / <30 做空) + 浮亏 > 1%
2. RSI 与持仓方向背离（做多但RSI下行至<50，做空但RSI上行至>50）
3. 收盘价跌破支撑 > 1%（做多）或突破阻力 > 1%（做空）
4. 量比持续 < 1.0（三根K线都缩量）
5. 铁律违反：做多不在支撑位附近，做空不在压力位附近

**过早入场检测（4信号，满足≥2条触发）：**
1. 入场价距S/R > 3%（中间地带入场）
2. 入场时量比 < 2.5x
3. R3引擎判定过早
4. 入场RSI极端值

**三轮确认门控：** 同一持仓连续3轮被标记才发飞书预警

### 2.2 现有API

| API | 用途 |
|-----|------|
| `GET /api/backtest/early_exit_status` | 前端轮询当前扫描结果 |
| `POST /api/backtest/early_exit_tasks` | 执行平仓/干预 |
| `_auto_early_exit_loop()` | 后台每5分钟扫描线程 |

### 2.3 现有问题

1. **无历史预警准确率记录** — 不知道预警后价格是否真的反转了
2. **多时线S/R位置不可见** — 看不到持仓时各时线支撑/压力位置
3. **干预动作单一** — 只有"通知"，没有分级干预策略
4. **无法交叉校验** — 多个机器人持同一交易对时信号不能共享

### 3.2 干预决策矩阵

#### 3.2.1 过早入场（方向正确，时机错误）

| 综合评分 | 持仓状态 | 动作 |
|---------|---------|------|
| 60-79分 | 方向对 | 🟡 保持观察，等信号确认 |
| 40-59分 | 方向对但弱 | 🟡 减半仓，设置-1.5%补仓线 |
| <40分 | 方向对但弱 | 🔴 半仓，触发-1.5%补仓线时加倍补（均摊成本） |

**补仓策略（过早入场专用）：**
- 首次补仓：亏损达到-1.5%时，加倍补（1.5x仓）
- 目标：拉平平均成本价
- 限制：最多补2次
- 逻辑：方向对但入场早，先占坑，成本均摊后小幅上涨即盈利

#### 3.2.2 方向错误

| 综合评分 | 持仓状态 | 动作 |
|---------|---------|------|
| 50-79分 | 方向错但强势 | 🟡 半仓，等反弹平 |
| <50分 | 方向错且弱势 | 🔴 **立即全仓平** |

#### 3.2.3 方向错误 + 垃圾仓位（评分<40 + 多重S/R违反）

- 🔴 **立即全仓平**
- 同时发反手信号（可选，需人工确认）
- 记录为"高危仓位"，下次入场同一交易对时给出警告

### 3.3 多时线S/R交叉校验

```
交易对信号共享原则：
- 同一交易对在多个机器人上持仓 → OHLCV数据共享获取
- 交易所不同但K线可交叉使用（OKX/Gate的BTC数据基本一致）
- 优先用机器人自有数据，数据不够时借用同交易对其他机器人的数据

置信度等级：
  🟢 高置信：3+个时线共振（同侧S/R）
  🟡 中置信：2个时线共振
  🔴 低置信：1个时线或无明显S/R
```

### 3.4 补仓均摊公式

```
初始仓位: 1x
首次补仓: 1.5x  (亏损-1.5%时执行)
二次补仓: 2x    (亏损-2.5%时执行)

平均成本 = (入场1价×1 + 补1价×0.5) / 1.5
均摊后微涨+0.8%即盈利
```

### 3.5 预警准确率追踪

每条预警记录：
```
{
  ts: "18:35",
  port: 9095,
  pair: "DOT/USDT",
  type: "方向错误",
  rounds_triggered: 3,
  action_taken: "半仓平",
  action_price: 1.230,
  price_after_10min: 1.215,  ← 10分钟后价格（验证是否反转）
  reversed: true,               ← 预警准确率统计
  correct: true
}
```

### 4.2 数据源优先级

```
同一交易对多机器人持仓时的 OHLCV 获取：
  1. 优先使用持仓机器人的 exchange_id 获取（自己获取自己用）
  2. 若获取失败，借用其他机器人的 exchange 获取（OKX数据 OKX共享）
  3. Gate 机器人可借用 OKX 同交易对数据（K线本质相同）

限制：每次扫描最多同时获取 N 个 OHLCV（N=5），避免触发交易所限速
```

### 4.3 补仓执行

```
补仓触发条件：
  - 过早入场 + 评分<60 + 亏损达到-1.5%
  - 当前亏损额 × 1.5 ≤ 补仓后预估最大亏损
  - 未达到补仓上限（最多2次）

补仓计算：
  补仓量 = 当前持仓量 × 0.5
  补仓价格 = 当前价格（按市价）

执行：
  POST /api/monitor/add_position
  {
    "port": 9097,
    "pair": "DOGE/USDT:USDT",
    "side": "long",
    "amount": "补仓量",
    "leverage": 10
  }
```

## 五、准确率追踪机制

```
每条预警记录保存以下字段用于统计：

trigger_ts      预警触发时间
action_ts      干预执行时间
action_type    干预类型（half_close / full_close / dca / observe）
price_at_trigger  触发时价格
price_after_5min  5分钟后价格
price_after_10min 10分钟后价格
price_after_30min 30分钟后价格
reversed       是否反转（价格向有利方向移动0.5%+）
correct        预警是否正确（结合持仓最终盈亏判断）

准确率 = 正确的预警数 / 实际触发的干预数
```
