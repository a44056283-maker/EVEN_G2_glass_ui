## 三、检测规则（增强版）

### 3.1 方向错误检测（5选3触发）

| # | 信号 | 做多条件 | 做空条件 |
|---|------|---------|---------|
| 1 | RSI极端区背离 | RSI>70 + 浮亏>1% | RSI<30 + 浮亏>1% |
| 2 | RSI方向背离 | 做多但RSI<50且下行 | 做空但RSI>50且上行 |
| 3 | S/R破位 | 跌破支撑>1% | 突破阻力>1% |
| 4 | 缩量确认 | 连续3根K线量比<1.0 | 连续3根K线量比<1.0 |
| 5 | 铁律违反 | 做多不在支撑附近 | 做空不在压力附近 |

**触发条件：满足≥3条（满分100，每条20分，≥60分触发）**

---

### 3.2 过早入场检测（4选2触发）

| # | 信号 | 阈值 |
|---|------|------|
| 1 | 入场距S/R距离 | >3%（中间地带） |
| 2 | 入场量比 | <2.5x |
| 3 | R3引擎判定 | 过早信号 |
| 4 | 入场RSI | 非极端值入场 |

**触发条件：满足≥2条（满分100，每条25分，≥50分触发）**

---

### 3.3 分级轮次门控（V2.0核心改进）

| 评分 | 风险等级 | 轮次要求 | 说明 |
|------|---------|---------|------|
| <40分 | 🔴 高危 | **1轮触发** | 立即预警，等不及3轮 |
| 40-59分 | 🟠 中高危 | **2轮触发** | 快速响应 |
| ≥60分 | 🟡 中风险 | **3轮触发** | 保持观察 |

**设计理由**：低分高危仓位的危害远大于高分仓位，3轮门控对高危仓位太慢，可能错过最佳干预时机。

## 五、动态补仓策略（V2.0核心）

### 5.1 ATR动态阈值（替代固定-1.5%/-2.5%）

> 固定百分比不科学：DOT波动大，SOL中等，BTC稳定，用同一阈值会导致DOT补在半山腰、BTC跌1.5%已是强信号。

**计算公式：**
```
ATR(14) = 最近14根K线的 True Range 平均值
首次补仓触发 = 亏损额 ≥ 0.5 × ATR(14)
二次补仓触发 = 亏损额 ≥ 1.0 × ATR(14)
```

**各交易对ATR参考（示例）：**
| 交易对 | 波动率 | ATR(14)示例 | 首次补仓线 | 二次补仓线 |
|--------|--------|-----------|-----------|-----------|
| BTC/USDT | 低 | $150 | -$75 | -$150 |
| ETH/USDT | 中 | $15 | -$7.5 | -$15 |
| SOL/USDT | 高 | $0.8 | -$0.4 | -$0.8 |
| DOT/USDT | 极高 | $0.05 | -$0.025 | -$0.05 |
| DOGE/USDT | 极高 | $0.003 | -$0.0015 | -$0.003 |

### 5.2 补仓层级公式

```
层数    触发条件              补仓比例    累计仓位
初始    入场                  1.0x        1.0x
L1补    亏损 ≥ 0.5×ATR        +0.5x       1.5x
L2补    亏损 ≥ 1.0×ATR        +0.5x       2.0x
```

**均摊成本计算：**
```
初始成本 = 入场价 × 1.0x
L1补后均摊 = (入场价 × 1.0 + 补仓价 × 0.5) / 1.5
L2补后均摊 = (入场价 × 1.0 + 补1价 × 0.5 + 补2价 × 0.5) / 2.0
```

**盈利触发点（L2补后）：**
```
DOT: 均摊后微涨+0.8%即盈利
SOL: 均摊后微涨+1.2%即盈利
BTC: 均摊后微涨+0.5%即盈利
```

### 5.3 补仓限制规则

| 规则 | 值 | 说明 |
|------|-----|------|
| 最大补仓次数 | 2次 | L1 + L2，最多2次 |
| 最大总持仓 | 3层 | 初始+2次补仓 |
| 补仓间隔 | ≥30分钟 | 防止连续触发 |
| 补仓冷却期 | 30分钟 | 防止频繁补 |
| 补仓后止损 | 均摊价的-5% | 补仓后新止损线 |

### 5.4 补仓触发决策树

```
持仓触发预警
    │
    ├── 过早入场 ──── 评分<60 ── 亏损≥0.5×ATR ──→ L1补仓
    │                                         │
    │                                    亏损≥1.0×ATR ──→ L2补仓（全平线：均摊-5%）
    │                                         │
    │                                    补仓后RSI极端 ──→ 全平
    │
    └── 方向错误 ─────────────────────────────→ 全平（不补仓）
```

- 14天内无任何DOT开仓记录

- 手动解锁（需人确认）

## 七、多时线S/R交叉校验（V2.0增强）

### 7.1 数据源优先级

```
优先级1：机器人自有exchange获取OHLCV（自己数据自己用）
优先级2：同交易所同交易对的其他机器人借用（OKX→OKX）
优先级3：跨交易所借用（Gate→OKX），K线形态相同可用
```

**限速保护：**
- 每次扫描最多同时获取5个交易对的OHLCV
- 每个交易对最多3个时间框架（15m/30m/1h）
- 交易所API限速：OKX 120次/2分钟，Gate 180次/分钟

### 7.2 置信度等级

| 等级 | 条件 | 信号强度 |
|------|------|---------|
| 🟢 高置信 | 3+个时线共振（同侧S/R） | 强，可立即行动 |
| 🟡 中置信 | 2个时线共振 | 中，需结合评分决策 |
| 🔴 低置信 | 1个时线或无明显S/R | 弱，保持观察 |

### 7.3 共振时线判定

```
做多信号（支撑共振）：
  15m支撑 + 30m支撑 同时存在 → 2线共振
  +1h支撑 → 3线共振

做空信号（压力共振）：
  15m压力 + 4h压力 同时存在 → 2线共振
```

## 八、准确率追踪机制（V2.0完整版）

### 8.1 预警记录数据模型

```python
class AlertRecord:
    alert_id: str           # UUID，唯一标识
    trigger_ts: float         # 预警触发时间（Unix timestamp）
    port: int               # 机器人端口
    pair: str               # 交易对
    direction: str          # LONG/SHORT
    alert_type: str         # "wrong_direction" / "premature_entry"
    score: int              # 触发时评分（0-100）
    rounds_triggered: int   # 触发轮次（1/2/3）
    action_taken: str       # "observe"/"half_close"/"full_close"/"dca_l1"/"dca_l2"
    action_ts: float | None # 干预执行时间
    action_price: float     # 干预执行价格
    entry_price: float     # 原始入场价
    price_at_trigger: float # 触发时价格

    # 验证字段（触发后填充）
    price_after_5min: float | None
    price_after_10min: float | None
    price_after_30min: float | None
    price_at_close: float | None    # 平仓时价格
    realized_pnl: float | None      # 最终实际盈亏

    # 判定字段
    reversed_5min: bool | None     # 5分钟内是否反转
    reversed_10min: bool | None    # 10分钟内是否反转
    correct: bool | None           # 预警是否正确（综合判定）
```

### 8.2 准确率计算公式

```
准确率 = 正确的预警数 / 实际触发的干预数

正确预警定义：
  ✅ 正向正确：干预后价格向有利方向移动 ≥0.5×ATR，且最终平仓盈利
  ✅ 负向正确：预警方向正确但未干预，最终持仓亏损（预警有效）
  ❌ 误报：预警触发但价格未反转，最终持仓持平或盈利（预警无效）
```

### 8.3 准确率报表SQL

```sql
-- 按交易对准确率
SELECT pair,
       COUNT(*) as total_alerts,
       SUM(CASE WHEN correct = 1 THEN 1 ELSE 0 END) as correct_count,
       ROUND(SUM(CASE WHEN correct = 1 THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 1) as accuracy_pct
FROM alert_records
WHERE trigger_ts > datetime('now', '-7 days')
GROUP BY pair
ORDER BY accuracy_pct ASC;

-- 按预警类型准确率
SELECT alert_type,
       COUNT(*) as total,
       SUM(CASE WHEN correct = 1 THEN 1 ELSE 0 END) as correct,
       ROUND(SUM(CASE WHEN correct = 1 THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 1) as accuracy
FROM alert_records
GROUP BY alert_type;

-- 按动作类型准确率
SELECT action_taken,
       COUNT(*) as total,
       AVG(realized_pnl) as avg_pnl,
       ROUND(SUM(CASE WHEN correct = 1 THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 1) as accuracy
FROM alert_records
WHERE action_taken != 'observe'
GROUP BY action_taken;
```

-- 预警记录表
CREATE TABLE alert_records (
    alert_id TEXT PRIMARY KEY,

    dca_id TEXT PRIMARY KEY,

## 十四、天禄补充的10个优化点（额外增加）

1. **相邻K线过滤**：预警触发后，下一个5分钟K线不重复触发（冷却机制）
2. **跨机器人信号共享**：同一交易对多个机器人时，OHLCV数据合并使用，减少API调用
3. **预警声音提醒**：高危预警（评分<40）触发时，前端播放提示音
4. **飞书预警分级推送**：高危立即推，中危等2轮，低危等3轮，避免信息轰炸
5. **误报自学习**：连续3次误报的预警类型，自动加严该类型的触发门槛+10分
6. **补仓线动态调整**：若补仓后价格继续下跌，ATR自动放大1.2倍（波动加剧）
7. **平仓价格记录**：干预执行后，记录实际成交价格（可能有滑点）
8. **预警时线标记**：记录触发预警时价格所在时线（方便回溯分析）
9. **交易所分组显示**：OKX机器人和Gate机器人分组显示，避免混淆
10. **实时S/R热力图**：根据多时线S/R密度，显示价格区域的热力程度
