# 吏部改造方案

**日期：** 2026-03-30  
**负责人：** 吏部尚书  
**参考框架：** agent-team-orchestration

---

## 一、改造背景

吏部原定位为「人事执行者」，以接收尚书省任务并亲自执行为主。随着省内 Agent 数量增长（目前已覆盖六部+中书省/尚书省/门下省），吏部面临两个核心矛盾：

1. **任务数量超出单人执行能力**：需同时管理多个 Agent 的任务生命周期
2. **角色分工模糊**：Builder/Reviewer/Ops 角色未明确界定，导致任务流向混乱

## 二、改造目标

将吏部从「执行者」升级为 **Orchestrator（编排者）**，实现：

- 任务路由自动化：根据任务类型自动分派到合适的 Agent
- 状态追踪可视化：通过看板实时反映每项任务所处阶段
- 交接协议结构化：所有跨 Agent 的交付使用标准 Handoff 格式
- 质量门禁前置：Review 阶段明确审点，不合格打回

## 三、核心改造内容

### 1. 角色体系重构

| 原角色 | 新角色 | 说明 |
|--------|--------|------|
| 吏部（执行） | **Orchestrator（编排者）** | 路由、追踪、优先级决策 |
| 各部署Agent | **Builder（执行者）** | 接收任务、执行、交付 |
| 尚书省/中书省 | **Reviewer（审核者）** | 质量审核、通过/打回 |
| 工部/门下省 | **Ops（运维）** | 部署、环境、监控 |

### 2. 任务生命周期标准化

```
Inbox → Assigned → In Progress → Review → Done
                  ↘ Blocked ↗         ↘ Failed
```

每个状态变更须通过 `kanban_update.py` 更新看板。

### 3. Handoff 消息标准化

所有跨角色交付使用标准结构（见 SOUL.md），包含：
- 任务基本信息（ID、标题、优先级、截止时间）
- 上下文摘要（已完成部分、待处理项）
- 交付物清单
- 注意事项与特殊约束
- 上游依赖

### 4. 分派决策规则

| 任务类型 | 分派方向 |
|----------|----------|
| 代码开发 | → 工部（Builder） |
| 安全审计 | → 刑部（Builder）+ 尚书省（Reviewer） |
| 文档编写 | → 礼部（Builder） |
| 数据处理 | → 户部（Builder） |
| 测试验证 | → 兵部（Builder） |
| 部署运维 | → 工部（Ops）+ 门下省 |
| 跨部任务 | → 拆解，吏部追踪依赖关系 |

## 五、过渡期注意事项

1. **双向兼容**：改造初期（1-2周），吏部仍保留对简单任务的直接执行能力
2. **尚书省确认**：重大任务分派须上报尚书省备案，防止越权
3. **看板数据迁移**：历史任务数据无需迁移，新任务一律采用新生命周期
4. **Reviewer 授权**：尚书省/中书省需确认接受 Review 角色，并配置相应权限
