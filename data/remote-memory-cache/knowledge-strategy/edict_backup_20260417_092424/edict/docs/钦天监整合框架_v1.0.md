**核心原则**：
1. **不直接干预** — 所有操作通过飞书提案审批，半自动执行
2. **情报驱动** — 基于真实市场数据，不是占位符
3. **分层决策** — 6层规则从P0(风控)到P3(事件)，按紧急度排序
4. **仅提供信号** — 最终决策权在审批者（爸），系统只做信息提供和执行

## 三、6层规则体系（与干预框架v1.0对齐）

### P0 · 风控层（Risk）
**触发条件**：绝对红线，系统自动提交最高优先级审批

| 规则ID | 名称 | 阈值 | 动作 |
|--------|------|------|------|
| R001 | 单日亏损熔断 | 日盈亏 ≤ -5% | `freeze` 全市场冻结 |
| R002 | 单周亏损暂停 | 周盈亏 ≤ -15% | `pause` 暂停新开仓 |
| R003 | 最大持仓超时 | 持仓 > 48小时 | `force_close` 强制平仓 |
| R004 | 拖尾止损触发 | 持仓未实现亏损 > -2% 且超24h | `force_stop_loss` 强制止损 |

### P1 · 市场结构层（Market）
**触发条件**：低波动/无趋势/价格偏离时阻止开仓

| 规则ID | 名称 | 阈值 | 动作 |
|--------|------|------|------|
| M001 | ATR波动率过滤 | ATR < 1.5% | `block_entry` 阻止做多/做空 |
| M002 | ADX趋势强度过滤 | ADX < 25 | `block_entry` 阻止追趋势 |
| M003 | EMA50方向过滤 | 价格 < EMA50 | `direction_bias` 只允许做空 |
| M004 | VWAP偏离过大 | 价格偏离VWAP > 1% | `block_entry` 阻止开仓 |

### P1 · 资金流层（Flow）
**触发条件**：资金费率异常/OI剧变时预警或阻止

| 规则ID | 名称 | 阈值 | 动作 |
|--------|------|------|------|
| F001 | BTC.Dominance强势 | BTC.D > 52% | `direction_bias` 禁止做空 |
| F002 | 资金费率过高 | \|费率\| > 0.05%/8h | `block_entry` 阻止开仓 |
| F003 | OI合约持仓量剧变 | OI变化 > ±20% | `alert` 预警+观察 |
| F004 | 净流出预警 | 主力净流出信号 | `block_short` 禁止做空 |

### P2 · 技术信号层（Tech）
**触发条件**：RSI极值/布林带收口/量能爆发

| 规则ID | 名称 | 阈值 | 动作 |
|--------|------|------|------|
| T001 | MACD底背离 | 价格新低但MACD未新低 | `signal_boost` 增强做多信号 |
| T002 | RSI极值 | RSI < 30 或 RSI > 70 | `signal_boost` 或 `confirm_entry` |
| T003 | 布林带收口 | 带宽 < 80%历史分位 | `watch_breakout` 关注突破方向 |
| T004 | 量能爆发 | 量比 > 3x | `confirm_breakout` 确认入场信号 |

### P2 · 情绪层（Sentiment）
**触发条件**：FG极度恐慌/贪婪时逆向提示

| 规则ID | 名称 | 阈值 | 动作 |
|--------|------|------|------|
| S001 | 极度恐慌逆向 | FG ≤ 10 | `reverse_signal` 逆向买入机会提示 |
| S002 | 恐惧禁止做空 | FG < 25 | `block_short` 禁止做空 |
| S003 | 贪婪禁止做多 | FG > 75 | `block_long` 禁止做多 |

### P3 · 事件驱动层（Event）
**触发条件**：ETF到期/大户异动/安全事件

| 规则ID | 名称 | 阈值 | 动作 |
|--------|------|------|------|
| E001 | ETF期权到期周 | 每月第四周五 | `reduce_position` 减仓提示 |
| E002 | 大户异动 | WhaleAlert API信号 | `alert` 飞书预警 |
| E003 | 交易所安全事件 | 情报API | `emergency_exit` 紧急撤退审批 |
| E004 | 稳定币流动性预警 | 链上数据 | `alert` 观察提示 |

## 四、与现有系统的关系

```
现有系统                    新钦天监
─────────────────────────────────────────────
兵部监控台 (7892)      →  钦天监数据 → 整合进同一UI
干预框架_v1.0         →  规则引擎   → 重写数据层，对接真实API
bingbu_intervention_   →  执行层     → 保持不变，飞书审批通道不变
  watchdog.py
动态干预Tab (7891)     →  展示层     → 新增钦天监信号汇总
```

**不冲突**：
- 钦天监只生成情报和信号，不直接操作机器人
- 所有干预仍通过现有 `bingbu_card_builder.py` 飞书提案通道
- 风控审批链：钦天监采集 → 规则引擎评估 → 飞书提案 → 爸审批 → 执行

## 八、审批流：爸的体验

当钦天监触发规则时，爸在飞书收到卡片：
```
┌─────────────────────────────────────────────────────┐
│ 🚨 钦天监 · 规则触发                               │
│                                                     │
│ 🔴 P0 · R001 单日亏损熔断                          │
│                                                     │
│ 当前日盈亏: -5.2%                                   │
│ 阈值: ≤ -5.0%                                       │
│ 建议动作: 全市场冻结                                │
│                                                     │
│ [批准执行]  [查看详情]  [忽略]                      │
└─────────────────────────────────────────────────────┘
```

爸点击批准 → 系统执行 `freeze` → 飞书收到执行结果通知
