## 一、三层监控架构优化方案

### 1.1 当前架构问题

现有架构存在以下问题：

| 问题 | 影响 | 优先级 |
|------|------|--------|
| 三层检查混在 port_scan.py，未分层隔离 | 故障定位困难 | P0 |
| L3 API业务检查仅测ping，未覆盖持仓/余额 | 无法发现业务层异常 | P0 |
| 无统一健康状态聚合出口 | 外部系统难以查询 | P1 |
| 看门狗重启逻辑分散 | 维护困难，可能重复重启 | P1 |

### 1.2 优化方案

```
┌──────────────────────────────────────────────────────────────────┐
│                    三层监控架构优化                                  │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐      │
│  │  L1 进程层   │───▶│  L2 端口层   │───▶│  L3 业务层   │      │
│  │  (pgrep)     │    │  (curl)      │    │  (API指标)   │      │
│  └──────────────┘    └──────────────┘    └──────────────┘      │
│        │                   │                   │                  │
│        ▼                   ▼                   ▼                  │
│  进程PID检查          HTTP 200检查        业务指标检查             │
│  CPU/内存            响应时间            余额/持仓/订单           │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │              健康状态聚合服务 (9099/health)                │   │
│  │  - /health/L1  L1层状态                                │   │
│  │  - /health/L2  L2层状态                                │   │
│  │  - /health/L3  L3层状态                                │   │
│  │  - /health/all 聚合总体状态                             │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 1.3 分层职责

| 层级 | 检查内容 | 检查频率 | 失败动作 | 恢复动作 |
|------|---------|---------|---------|---------|
| L1 进程层 | 进程PID存在、CPU使用率、内存占用 | 30秒 | 立即重启 + CRITICAL告警 | INFO通知 |
| L2 端口层 | HTTP响应状态码、连接建立时间 | 60秒 | 重启 + WARNING告警 | INFO通知 |
| L3 业务层 | 余额>0、持仓一致性、API响应时间 | 120秒 | WARNING告警 | INFO通知 |

### 1.4 改进收益

| 改进点 | 改善效果 |
|--------|---------|
| 分层独立检查 | 故障定位时间从 ~10分钟 降至 ~1分钟 |
| L3业务层新增 | 可发现"进程存活但业务卡死"的边缘故障 |
| 统一聚合出口 | 便于外部系统(Nagios/Zabbix/Prometheus)集成 |
| 重启逻辑集中 | 避免多进程竞争重启导致的震荡 |

## 二、告警分级优化方案

### 2.1 当前问题

| 问题 | 现状 |
|------|------|
| 告警粒度粗 | 仅CRITICAL/WARNING/INFO三级 |
| 告警无升级机制 | 不区分白天/夜间/节假日响应要求 |
| 告警无收敛 | 同一故障重复告警，无法抑制 |
| 告警无确认 | 无法追踪告警处理状态 |

### 2.2 优化方案：五级告警 + 升级机制

```
告警分级定义
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  P0 CRITICAL (红色)
  ├── 定义: 服务彻底不可用，或交易机器人失控
  ├── 响应: 立即处理（不限时间）
  ├── 通知: 飞书@所有人 + 电话
  └── 示例: 进程死亡、端口超时>30s、强平风险

  P1 HIGH (橙色)
  ├── 定义: 核心功能受损，但有备用路径
  ├── 响应: 30分钟内处理
  ├── 通知: 飞书@全体 + 工部主任
  └── 示例: 单Bot无响应、API响应>5s

  P2 MEDIUM (黄色)
  ├── 定义: 性能下降，不影响核心功能
  ├── 响应: 2小时内处理
  ├── 通知: 飞书工部群
  └── 示例: 内存>80%、响应>2s

  P3 LOW (蓝色)
  ├── 定义: 警告/预防性提示
  ├── 响应: 工作时间处理
  ├── 通知: 工部日志
  └── 示例: L3指标接近阈值、Bot胜率下降

  P4 INFO (绿色)
  ├── 定义: 正常事件记录
  ├── 响应: 无需处理
  ├── 通知: 仅日志
  └── 示例: 服务恢复、交易平仓
```

### 2.3 告警升级机制

```
告警发出后未被确认？触发升级：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  P0 CRITICAL
  └── 5分钟无确认 → 自动升级为电话呼叫
  └── 15分钟无确认 → 通知太子(皇上)

  P1 HIGH
  └── 15分钟无确认 → @工部主任
  └── 30分钟无确认 → 升级为P0

  P2 MEDIUM
  └── 2小时无确认 → 升级为P1
```

### 2.4 告警收敛（抑制策略）

| 场景 | 抑制规则 |
|------|---------|
| 同一端口反复重启 | 5分钟内只发1次CRITICAL，后续抑制 |
| 网络抖动 | 连续3次失败才触发告警（防误报） |
| 计划内维护 | 维护窗口期间告警静默 |
| 恢复后 | 自动发送P4恢复通知，替换原有告警 |

### 2.5 告警通知模板

```
【{级别标签}】{服务名} · {告警类型}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
时间: {timestamp}
端口: {port}
异常: {symptom}
持续: {duration}
建议: {action}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
工部告警系统 · {消息序号} · 如已处理请忽略
```

## 三、console_server.py 健康检查优化建议

### 3.1 当前健康检查现状

现有 `/health` 端点结构：

```json
{
  "status": "healthy|degraded|unhealthy",
  "layers": {
    "L1_process": {"status": "ok|error", "pid": 12345},
    "L2_memory": {"status": "ok|warn|error", "usage_percent": 65.3},
    "L3_api": {"status": "ok|error", "avg_response_time": 0.182}
  }
}
```

**现有端点**：
- `GET /health` - 总体状态
- `GET /health/liveness` - K8s存活探测
- `GET /health/readiness` - K8s就绪探测

### 3.2 发现的问题

| 问题 | 说明 | 优先级 |
|------|------|--------|
| L3 API检查仅测ping | 无法反映实际业务健康（如余额为0但API正常）| P0 |
| 无 Bot 级别细分状态 | 9090-9093 任一Bot挂了，总体状态可能仍healthy | P1 |
| 无历史状态追踪 | 无法看到健康状态变化趋势 | P1 |
| /health 无认证 | 任何人可查询内部状态 | P2 |
| readiness未检查Bot | 就绪探测未考虑Bot是否正常 | P1 |

### 3.3 优化建议

#### 建议一：L3 业务层增强（最高优先级）

**问题**：当前L3只测API ping，无法反映"进程活着但业务已死"的情况。

**优化方案**：在 `/health/L3` 增加业务层检查：

```python
# 新增 /health/L3 端点（可选）
# 检查内容：
# 1. 余额检查：任一Bot余额<10U → WARNING
# 2. 持仓异常：is_open=1但exit_reason=liquidation → CRITICAL
# 3. API响应时间：>3s → WARNING，>10s → CRITICAL
# 4. 交易对超时：任一交易对>30分钟无新K线 → WARNING
```

#### 建议二：Bot 级别状态细分

**问题**：现有状态是聚合的，无法定位具体哪个Bot出问题。

**优化方案**：在 `/health` 响应中增加 per-bot 状态：

```json
{
  "status": "degraded",
  "bots": {
    "9090": {"status": "ok", "balance": 1121.68, "positions": 4, "api_time_ms": 142},
    "9091": {"status": "ok", "balance": 1255.39, "positions": 3, "api_time_ms": 138},
    "9092": {"status": "degraded", "balance": 1248.41, "positions": 4, "api_time_ms": 5200, "issue": "response_slow"},
    "9093": {"status": "critical", "balance": 1367.18, "positions": 4, "api_time_ms": 0, "issue": "eth_position_loss_23pct"}
  },
  "layers": { ... }
}
```

#### 建议三：健康状态历史追踪

**问题**：无法看到健康状态变化趋势。

**优化方案**：增加 `/health/history` 端点，返回最近1小时状态变化：

```json
{
  "history": [
    {"timestamp": "2026-03-30T09:35:00Z", "status": "healthy", "event": null},
    {"timestamp": "2026-03-30T09:30:00Z", "status": "degraded", "event": "9092响应慢"},
    {"timestamp": "2026-03-30T09:28:00Z", "status": "healthy", "event": "自动恢复"},
    {"timestamp": "2026-03-30T09:25:00Z", "status": "unhealthy", "event": "9093 ETH仓位异常"}
  ]
}
```

#### 建议四：增加认证保护

**问题**：`/health` 端点无认证，信息泄露风险。

**优化方案**：
- 添加 `X-Health-Token` 请求头验证
- Token 从配置文件读取
- 提供 `/health/open` 端点（仅返回 status，不含详情）

#### 建议五：readiness 探测增加 Bot 检查

**问题**：当前 readiness 只检查内存，不检查Bot状态。

**优化方案**：修改 `/health/readiness` 逻辑：

```python
# readiness 应同时满足：
# 1. 内存 < threshold
# 2. 4个Bot至少2个在线（允许降级运行）
# 3. 无P0告警状态
```

### 3.4 建议实施优先级

| 优先级 | 建议 | 工作量 | 风险 |
|--------|------|--------|------|
| P0 | L3业务层增强（余额/持仓检查） | 中 | 低 |
| P0 | Bot级别状态细分 | 小 | 低 |
| P1 | readiness增加Bot检查 | 小 | 低 |
| P1 | 健康状态历史追踪 | 中 | 中（需存储） |
| P2 | 健康端点认证保护 | 小 | 低 |
| P2 | 独立 /health/L3 端点 | 中 | 低 |
