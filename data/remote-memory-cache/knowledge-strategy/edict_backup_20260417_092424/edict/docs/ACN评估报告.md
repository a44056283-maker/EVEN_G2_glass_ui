# ACN 评估报告

**评估日期：** 2026-03-30  
**评估者：** 吏部尚书  
**评估对象：** Agent Collaboration Network（ACN）  
**版本：** v0.4.5（acnlabs/ACN）

---

## 一、ACN 平台概述

ACN（Agent Collaboration Network）是一个开源基础设施，提供 AI Agent 的：
- **注册与发现**：通过 API 注册，获取 agent_id + api_key
- **能力检索**：按 skill 搜索在线 Agent
- **任务协作**：创建/接受/提交/审核任务的完整生命周期
- **消息通信**：Agent 间直接消息和广播
- **子网隔离**：私有子网（subnet）分组
- **链上身份**：ERC-8004 NFT（Base 主网）永久身份

## 二、价值评估

### 2.1 核心价值 ✅

| 价值点 | 说明 | 评分 |
|--------|------|------|
| Agent 发现 | 通过 skill 标签发现省内 Agent，无需手动维护 | ⭐⭐⭐⭐ |
| 任务协作 | 标准化的 create/accept/submit/review 流程 | ⭐⭐⭐⭐ |
| 状态追踪 | 任务生命周期完整记录 | ⭐⭐⭐⭐ |
| 信任机制 | Escrow 接口支持有偿任务的资金托管 | ⭐⭐⭐ |
| 跨平台 | REST API + Python SDK，兼容性好 | ⭐⭐⭐⭐ |
| 开源 | MIT 许可证，可自托管 | ⭐⭐⭐⭐⭐ |

### 2.2 与省内系统的互补性

当前省内系统（Edict 三兄弟）：
- **三兄弟**：主Agent（天禄）+ 备用Agent（天福/天禧）— 冗余保障
- **看板**：任务追踪（`kanban_update.py`）
- **协作**：尚书省/吏部的层级调度

ACN 可补充：
- **跨 Agent 实时发现**：无需预先配置 endpoint
- **标准化任务协议**：Escrow + 多方审核
- **公开市场**：可引入外部 Agent 执行付费任务

## 三、风险评估 ⚠️

### 3.1 必须重视的风险

| 风险 | 级别 | 说明 |
|------|------|------|
| **API Key 泄露** | 🔴 高 | ACN_API_KEY 持有全部权限（发消息、接受任务、heartbeat） |
| **数据上传第三方** | 🟡 中 | Agent 信息、任务描述、消息内容均上传 acn-production.up.railway.app |
| **供应商锁定** | 🟡 中 | 自托管需要额外运维；依赖 railway.app 可用性 |
| **链上身份风险** | 🟡 中 | ERC-8004 需要钱包私钥，暴露即损失 |
| **任务数据留痕** | 🟡 中 | 任务历史、消息记录存于 ACN 服务器 |

### 3.2 数据安全细节

- **Agent Card**：包含 Agent 名称、描述、skill 列表、endpoint（公开）
- **任务描述**：创建的任务内容存储于 ACN（对于政务/机密任务有暴露风险）
- **消息内容**：直接消息和广播内容经过 ACN 服务器
- **无端到端加密**：ACN 服务器可读取所有内容

### 3.3 信任边界

ACN 的信任模型基于：
1. API Key 认证
2. Auth0 JWT（任务管理端点）
3. Escrow Provider（资金托管，第3方）

**吏部判断**：对于省内协作，ACN 作为内部工具使用时风险可控；对于外部协作/付费任务，Escrow 是必要的。

## 四、与现有系统整合可行性

### 4.1 与 Edict 看板系统的整合

| 维度 | Edict 看板 | ACN 任务 | 整合方案 |
|------|------------|----------|----------|
| 任务来源 | 尚书省下发 | ACN marketplace / 外部 | ACN 任务可导入 Edict 看板 |
| 生命周期 | 7态（Inbox→Done等） | 5态（open→done） | Edict 看板保留，ACN 作为外部补充 |
| 分派机制 | 吏部 Orchestrator | ACN 匹配 | 外部任务走 ACN，内部任务走 Edict |

**结论**：Edict 看板与 ACN 任务系统并行运作，通过 handoff 消息互通，不做深度集成。

### 4.2 与「三兄弟」系统的整合

三兄弟（天禄/天福/天禧）定位为高可用主备体系，ACN 可作为：
- **发现层**：外部 Agent 发现三兄弟的能力
- **消息层**：与外部 Agent 的通信通道
- **任务层**：外部付费/协作任务的接入点

**不推荐**：将三兄弟直接注册到 ACN（主Agent 信息暴露风险高）

### 4.3 与尚书省/吏部 Orchestrator 的整合

- 尚书省可授权 ACN 作为**任务来源之一**（外部Agent 接入）
- 吏部作为 Orchestrator，可通过 ACN API 发现外部协作 Agent
- ACN subnets 可模拟「部门子网」概念

## 五、建议方案

### 5.1 推荐策略：**审慎引入，内部禁用**

| 场景 | 建议 | 理由 |
|------|------|------|
| 省内 Agent 协作 | ❌ 不使用 ACN | 数据留痕风险；Edict 看板已满足需求 |
| 外部 Agent 协作 | ✅ 有限使用 | 通过 subnet 隔离，仅暴露必要信息 |
| 付费任务/众包 | ✅ 使用 | Escrow 机制有实际价值 |
| 链上身份 | ❌ 不注册 | 无实际需求；私钥管理成本高 |
| 自托管部署 | 🔄 长期考虑 | 可行，但需要额外运维投入 |

### 5.2 安全加固要求

如决定引入 ACN，必须：

1. **API Key 管理**：存储于环境变量，不写入任何配置文件
2. **最小权限注册**：Agent Card 仅披露必要信息，不暴露 endpoint 细节
3. **Subnet 隔离**：与外部 Agent 隔离在独立 subnet
4. **心跳最小化**：仅在有协作需求时启用，避免长期在线暴露
5. **任务内容脱敏**：创建任务时移除敏感信息（用占位符替代）

### 5.3 实施路线图

```
Phase 1（不引入）：
  - 继续使用 Edict 看板 + 尚书省/吏部 Orchestrator
  - 暂不注册 ACN

Phase 2（评估后引入）：
  - 仅用于外部付费/协作任务
  - 在 subnet 内注册，隔离于内网
  - API Key 由吏部尚书统一保管

Phase 3（自托管）：
  - 评估自托管 ACN 的运维成本
  - 如可行，将 ACN 完全内网化
```

## 六、最终结论

| 维度 | 评分 | 结论 |
|------|------|------|
| 技术价值 | ⭐⭐⭐⭐ | 任务协作框架成熟，开源可用 |
| 安全风险 | ⭐⭐ | 数据上传第三方，API Key 集中权限 |
| 整合可行性 | ⭐⭐⭐ | 与 Edict 并行运作，互补不替代 |
| 必要性 | ⭐⭐ | 省内协作现有系统已足够 |
| **综合推荐** | 🔄 **审慎评估后有限引入** | |

**核心建议**：ACN 的价值在于**外部协作和付费任务**，而非省内协作。在尚书省审批前，吏部建议：

1. **暂不将省内 Agent（天禄/天福/工部等）注册到 ACN**
2. **仅在使用外部 Agent 协作时启用 ACN**（新任务类型：跨组织）
3. **链上身份（ERC-8004）暂不考虑**
4. **持续关注 ACN 开源社区，自托管方案待评估**
